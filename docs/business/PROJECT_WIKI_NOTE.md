# Project Wiki Note

Every major project should maintain a standalone business/product wiki.

The wiki should capture original intent, user problems, business requirements, product logic, feature ideas, competitor examples, assumptions, decisions, open questions, and discussion notes.

The wiki should exclude implementation details such as code structure, APIs, database schema, deployment, testing internals, and build pipeline details.

Use the split below:

- Project wiki: why, what, product logic, user problems, market context, decisions.
- Technical docs: how, code, architecture, APIs, database, deployment, testing.

Brainstorming should happen around the wiki so useful thoughts become linked notes instead of getting lost in chats.

The wiki should behave like a small Wikipedia-style knowledge graph, with links between intent, problems, feature ideas, market examples, decisions, and future tasks.

For future work, combine the project wiki with IAC, ADS, and ODD:

- IAC: Impact Area Context.
- ADS: Autonomous Discovery Scope.
- ODD: Output Delivery Direction.

This gives future work persistent context plus clear task-level direction.
